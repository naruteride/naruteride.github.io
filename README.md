# HTML, CSS
1. **px, em, rem의 차이**:
   - **px (픽셀)**: 고정 크기 단위로, 화면 해상도에 따라 크기가 달라질 수 있습니다.
   - **em**: 상위 요소의 폰트 크기에 상대적으로 크기가 결정됩니다.
   - **rem**: 루트 요소(일반적으로 `<html>`)의 폰트 크기에 상대적으로 크기가 결정됩니다. 루트 요소의 크기를 조절하면 모든 요소에 일관된 영향을 미칩니다.

2. **반응형 웹에서 사용하는 단위**:
   - 반응형 웹에서는 주로 `%`, `em`, `rem`, `vw` (viewport width), `vh` (viewport height) 등을 사용합니다.

3. **vw와 vh**:
   - **vw (viewport width)**: 화면 너비의 백분율로, 뷰포트 너비에 상대적인 크기를 설정합니다.
   - **vh (viewport height)**: 화면 높이의 백분율로, 뷰포트 높이에 상대적인 크기를 설정합니다.

4. **반응형 브레이크 포인트**:
   - 반응형 브레이크 포인트는 디자인 및 콘텐츠 레이아웃이 잘 보이고 작동하는 크기를 기준으로 설정합니다. 주로 기기의 너비에 따라 레이아웃을 조정하는데 사용됩니다.

5. **CSS 선택자 우선순위**:
   - CSS 선택자 우선순위는 다음과 같은 순서대로 결정됩니다: 인라인 스타일 (`style` 속성), 아이디 선택자, 클래스/속성/가상 선택자, 요소 선택자. `!important` 규칙은 다른 모든 규칙을 무시하고 적용됩니다.

6. **비율을 유지하는 CSS 설정**:
   - `padding-bottom` 또는 `padding-top`을 사용하여 요소의 높이를 백분율로 설정하고, `width`를 설정하여 가로 크기를 조절합니다. 이렇게하면 요소가 페이지 크기에 관계없이 높이와 너비의 비율이 유지됩니다.

7. **Flexbox**:
   - Flexbox는 요소를 효율적으로 배치하고 정렬하는 CSS 레이아웃 모델입니다. 부모 요소에 `display: flex`를 설정하여 자식 요소를 정렬하고 공간 배분을 설정할 수 있습니다.

8. **float 동작**:
   - `float` 속성은 요소를 왼쪽 또는 오른쪽으로 떠서 텍스트 흐름 주위로 배치하는데 사용됩니다. 이전에 주로 레이아웃을 만드는 데 사용되었으나, Flexbox와 Grid 레이아웃으로 대체되었습니다.

9. **CSS Cascading**:
   - Cascading은 CSS 규칙이 충돌할 때 우선 순위를 결정하는 프로세스를 나타냅니다. 이는 세 가지 요소로 결정됩니다: 중요성, 특정도, 소스 순서.

10. **SCSS (Sass)**:
    - SCSS는 CSS의 확장 버전으로, 변수, 중첩 규칙, 믹스인 등의 기능을 제공하여 CSS 코드를 더 효율적으로 작성하고 관리하는 데 도움을 줍니다.

11. **position 속성**:
    - `position` 속성은 요소의 위치를 설정하는 데 사용됩니다. 주요 값으로 `static`, `relative`, `absolute`, `fixed`

 등이 있으며, 각각 다른 방식으로 요소를 배치합니다.

12. **margin과 padding**:
    - `margin`은 요소의 외부 여백을 설정하고, 요소 사이의 간격을 조정합니다.
    - `padding`은 요소의 내부 여백을 설정하고, 내용과 테두리 사이의 간격을 조정합니다.

13. **HTML 렌더링 도중 JavaScript 실행 멈춤**:
    - HTML 파서는 스크립트 태그를 만나면 DOM 구성이 멈추고 JavaScript를 다운로드하고 실행합니다. 따라서 JavaScript 실행이 끝날 때까지 페이지 렌더링이 중단될 수 있습니다.

14. **Box model**:
    - Box model은 CSS 박스를 설명하는 개념으로, 각 요소는 내용(content), 패딩(padding), 테두리(border), 여백(margin)으로 구성됩니다.

15. **Attribute vs. Property**:
    - HTML 요소의 속성(attribute)은 HTML 문서에서 정의된 속성을 나타내고, JavaScript의 `getAttribute` 메서드로 접근합니다. 반면에 속성은 DOM 요소의 프로퍼티(property)로 표현되며, JavaScript에서 직접 변경 및 접근 가능합니다.

16. **display 속성 값**:
    - `display` 속성은 요소의 표시 방법을 설정하는 데 사용됩니다. 주요 값으로 `block`, `inline`, `inline-block`, `none`, `flex`, `grid` 등이 있습니다.

17. **CSS 애니메이션 vs. JS 애니메이션**:
    - CSS 애니메이션은 CSS 속성을 사용하여 애니메이션을 정의하고 브라우저에 의해 처리됩니다. JS 애니메이션은 JavaScript 코드를 사용하여 애니메이션을 프로그래밍적으로 제어합니다.

18. **CSS in JS**:
    - CSS in JS는 JavaScript 안에 CSS 코드를 작성하고 스타일을 동적으로 생성하는 방식을 나타냅니다. 이를 통해 컴포넌트 기반 애플리케이션에서 스타일을 관리할 수 있습니다.

19. **시맨틱 마크업**:
    - 시맨틱 마크업은 HTML 요소를 문서 구조와 의미에 따라 사용하는 것을 의미합니다. 시맨틱 요소(예: `<header>`, `<nav>`, `<footer>`)는 콘텐츠의 의미를 더 명확하게 나타냅니다.

20. **HTML5 태그**:
    - HTML5는 새로운 시맨틱 태그(예: `<header>`, `<section>`, `<nav>`)를 도입하여 웹 문서의 의미와 구조를 향상시켰습니다.

21. **script 태그의 Async와 Defer**:
    - `async` 속성은 스크립트를 비동기적으로 다운로드하고 실행하며, 다운로드가 완료되면 즉시 실행합니다.
    - `defer` 속성은 스크립트를 비동기적으로 다운로드하고, HTML 파서가 페이지 파싱을 완료한 후에 스크립트를 실행합니다.

22. **가상 클래스**:
    - 가상 클래스는 특정 상태나 위치에 따라 요소에 스타일을 적용하는 데 사용됩니다. 예로는 `:hover`, `:active`, `:first-child` 등이 있습니다.

23. **Margin 병합**:
    - Margin 병합은 인접한 블록 요소의 상단 및 하단 마진이 병합되는 현상을 나타냅니다. 가장 큰 마진

 값으로 결합됩니다.














# 자바스크립트
자바스크립트는 웹 개발을 위한 프로그래밍 언어로, 웹 페이지를 동적으로 조작하고 상호작용하는 데 사용됩니다.

1. **변수 선언, 초기화, 할당의 차이점**:
   - **변수 선언**: 변수를 생성하고 메모리 공간을 확보합니다. (`var`, `let`, `const`로 선언)
   - **초기화**: 변수에 값을 할당하는 과정으로, 선언 후에 값을 대입합니다.
   - **할당**: 이미 선언된 변수에 새로운 값을 대입하는 것입니다.

2. **데이터 타입**:
   - 자바스크립트의 데이터 타입에는 원시 타입 (Primitive Types)과 참조 타입 (Reference Types)이 있습니다. 원시 타입은 숫자, 문자열, 불리언, null, undefined, symbol, bigint가 있고, 참조 타입은 객체, 배열, 함수 등이 있습니다.

3. **생성자**:
   - 생성자 함수는 객체를 생성할 때 사용되며, 객체를 초기화하고 반환하는 함수입니다. 일반적으로 생성자 함수의 이름은 대문자로 시작합니다.

4. **this**:
   - `this`는 현재 실행 중인 함수나 메서드에서 사용되며, 실행 컨텍스트에 따라 다르게 동작합니다. 주로 객체의 메서드 내에서 해당 객체를 가리키는 데 사용됩니다.

5. **call, apply, bind**:
   - 이들은 함수의 `this` 값을 변경하는 메서드입니다.
   - `call`과 `apply`는 함수를 호출하면서 `this`와 인수를 전달합니다.
   - `bind`는 함수를 호출하는 것이 아니라 새로운 함수를 생성하며, `this`와 인수를 설정합니다.

6. **콜백 함수**:
   - 콜백 함수는 다른 함수의 인수로 전달되고, 나중에 실행되는 함수입니다. 주로 비동기 작업에서 사용됩니다.

7. **콜백 지옥 해결 방법**:
   - 콜백 지옥은 콜백 함수가 중첩되어 가독성을 떨어뜨릴 때 발생합니다. 해결 방법으로는 Promise, async/await을 사용하거나 콜백을 모듈화하는 방법 등이 있습니다.

8. **Promise**:
   - Promise는 비동기 작업을 보다 간편하게 다룰 수 있도록 도와주는 객체로, 성공 또는 실패 시에 콜백을 호출합니다.

9. **Promise.all()**:
   - `Promise.all()`은 여러 개의 Promise를 병렬로 실행하고, 모든 Promise가 성공할 때까지 기다린 후 결과를 반환합니다.

10. **Promise vs. Callback**:
    - Promise는 비동기 코드를 보다 구조화하고 관리하기 쉽게 만듭니다. Callback은 중첩된 코드를 생성할 수 있어 가독성이 떨어질 수 있습니다.

11. **Async, Await**:
    - `async` 함수는 비동기 작업을 수행하고 `await` 키워드를 사용하여 Promise가 처리될 때까지 기다립니다.

12. **Promise vs. Async, Await**:
    - Promise는 콜백 지옥을 해결하고 비동기 코드를 더 간편하게 작성하도록 도와줍니다. Async, Await은 Promise를 더 읽기 쉽게 만들어줍니다.

13. **AJAX**:
    - AJAX(Asynchronous JavaScript and XML)는 비동기적으로 서버와 통신하기 위한 기술로, 웹 페이지를 새로 고치지 않고도 데이터를 불러올 수 있습니다.

14. **var, let, const 차이**:
    - `var`는 함수 스코프를 가지고 있고 호이스팅이 발생합니다.
    - `let`과 `const`는 블록 스코프를 가지고 있으며, `let`은 재할당 가능하고 `const`는 재할당이 불가능합니다.

15. **TDZ (Temporal Dead Zone)**:
    - TDZ는 `let`과 `const` 변수가 선언된 시점부터 초기화되기 전까지의 범위를 나타내며, 변수에 접근하면 에러가 발생합니다.

16. **함수 선언형 vs. 함수 표현식**:
    - 함수 선언형은 함수를 선언하고 변수에 할당합니다. 함수 표현식은 변수에 함수를 할당

합니다.

17. **이벤트 버블링과 캡처링**:
    - 이벤트 버블링은 하위 요소에서 상위 요소로 이벤트가 전파되는 것이고, 이벤트 캡처링은 상위 요소에서 하위 요소로 이벤트가 전파되는 것입니다.

18. **이벤트 위임**:
    - 이벤트 위임은 하위 요소에 대한 이벤트 리스너를 상위 요소에 하나만 등록하여 메모리 사용을 줄이고 성능을 향상시키는 패턴입니다.

19. **호이스팅**:
    - 호이스팅은 변수와 함수 선언이 스코프의 최상단으로 끌어올려지는 동작입니다.

20. **클로저 (Closure)**:
    - 클로저는 함수와 그 함수가 선언된 렉시컬 스코프 사이의 연결을 나타내며, 외부 스코프의 변수에 접근할 수 있도록 합니다.

21. **비동기**:
    - 비동기는 작업을 백그라운드에서 실행하고 결과를 기다리지 않고 다른 작업을 수행할 수 있는 방식을 의미합니다.

22. **동기 vs. 비동기**:
    - 동기 코드는 순차적으로 실행되며 작업이 완료될 때까지 대기합니다. 비동기 코드는 작업이 백그라운드에서 실행되며 기다리지 않고 다른 작업을 수행합니다.

23. **ES6 새로운 기능**:
    - 화살표 함수, 클래스, 템플릿 리터럴, 디스트럭처링 할당, 모듈 등이 추가되었습니다.

24. **Undefined, Null, Undeclared 비교**:
    - `undefined`는 변수가 선언되었지만 값이 할당되지 않았을 때의 상태입니다.
    - `null`은 변수에 의도적으로 빈 값을 할당한 상태입니다.
    - `undeclared`는 선언조차 되지 않은 변수를 나타냅니다.

25. **멀티 쓰레드처럼 사용**:
    - 자바스크립트는 단일 스레드 기반 언어이지만, 웹 워커(Worker)를 통해 멀티 스레드와 유사한 동작을 구현할 수 있습니다. 이를 통해 백그라운드에서 병렬 작업을 수행할 수 있습니다.

26. **이벤트 루프 (Event Loop)**:
    - 이벤트 루프는 비동기 코드의 실행을 관리하고 콜 스택, 큐, 마이크로태스크 큐를 사용하여 비동기 작업을 처리합니다.

27. **태스크 큐와 마이크로태스크 큐**:
    - 태스크 큐에는 비동기 작업이 순서대로 들어가며, 마이크로태스크 큐에는 Promise의 콜백이 들어갑니다.

28. **requestAnimationFrame**:
    - `requestAnimationFrame`은 브라우저에게 애니메이션을 수행하도록 지시하는 함수로, 최적화된 애니메이션을 구현할 때 사용됩니다.

29. **동기적 코딩**:
    - 비동기 코드를 동기적으로 처리하기 위해 Callback, Promise, Async/Await을 사용할 수 있습니다.

30. **map, forEach, reduce**:
    - `map`은 배열의 각 요소에 함수를 적용하고 새로운 배열을 반환합니다.
    - `forEach`는 배열의 각 요소에 함수를 적용하고 반환하지 않습니다.


    - `reduce`는 배열을 하나의 값으로 축소합니다.

31. **자바스크립트의 메모리 관리**:
    - 자바스크립트는 가비지 컬렉터를 사용하여 더 이상 사용되지 않는 메모리를 자동으로 회수합니다.

32. **클래스**:
    - 클래스는 객체 지향 프로그래밍에서 객체를 생성하기 위한 설계 템플릿으로, ES6에서 도입되었습니다.

33. **즉시 실행 함수 (IIFE)**:
    - IIFE는 선언과 동시에 즉시 실행되는 함수로, 변수 스코프를 제한하는 데 사용됩니다.

34. **엄격 모드**:
    - 'use strict'를 사용하여 엄격 모드를 활성화하면 더 엄격한 규칙을 적용하며 오류를 미리 방지합니다.

35. **콜 스택과 힙**:
    - 콜 스택은 함수 호출을 추적하는 데 사용되고, 힙은 동적으로 할당된 메모리를 관리하는 데 사용됩니다.

36. **Rest 연산자와 Spread 연산자**:
    - Rest 연산자 (`...`)는 함수 매개변수나 배열/객체를 나머지 요소로 수집합니다.
    - Spread 연산자 (`...`)는 배열/객체의 요소를 분해하여 전달합니다.

37. **제너레이터**:
    - 제너레이터는 함수의 실행을 일시 중단하고 나중에 다시 시작할 수 있는 기능을 제공합니다.

38. **이터러블과 이터레이터 프로토콜**:
    - 이터러블은 반복 가능한 객체를 나타내며, 이터레이터 프로토콜은 이터러블 객체와 순회하는 방법을 정의합니다.

# 타입스크립트
타입스크립트를 사용하는 이유:
1. **정적 타입 체크**: 타입스크립트는 변수, 함수, 객체 등의 요소에 명시적인 데이터 타입을 지정할 수 있어 코드의 안정성을 높입니다. 컴파일러가 코드를 검사하여 타입 관련 오류를 사전에 발견할 수 있습니다.

2. **코드 가독성 향상**: 타입 정보를 주석 형태로 코드에 포함시킬 수 있으므로 코드의 의도를 명확하게 전달할 수 있습니다. 이로써 코드 가독성을 높이고 유지보수를 용이하게 합니다.

3. **자동 완성 및 인텔리센스**: 개발 환경에서 타입 정보를 활용하여 자동 완성과 인텔리센스 기능을 제공하므로 개발자가 빠르고 정확하게 코드를 작성할 수 있습니다.

4. **리팩터링 지원**: 타입 정보를 가지고 있기 때문에 대규모 코드베이스에서의 리팩터링이 용이하며 안전하게 수행할 수 있습니다.

5. **효율적인 협업**: 타입 정보를 공유하고 문서화할 수 있어 여러 개발자 간의 협업이 원활하게 진행됩니다.

Type과 Interface의 차이점:
- **Type**: 주로 객체, 유니온, 인터섹션, 기본 타입 등을 정의하는데 사용됩니다. 타입은 확장이 불가능하며 두 개 이상의 타입을 합칠 때 유니온과 인터섹션을 사용합니다.

    ```typescript
    type Person = { name: string; age: number };
    type Point = { x: number; y: number };
    type PersonOrPoint = Person | Point;
    ```

- **Interface**: 객체의 구조(shape)를 정의하는데 주로 사용됩니다. 인터페이스는 확장이 가능하며, 클래스에서 구현할 수 있습니다.

    ```typescript
    interface Person {
      name: string;
      age: number;
    }
    
    class Student implements Person {
      constructor(public name: string, public age: number) {}
    }
    ```

제네릭:
- 제네릭은 타입 또는 함수에 대해 추상적인 타입을 사용할 수 있게 해줍니다. 즉, 타입이나 함수의 입력 및 출력 타입을 파라미터화하여 재사용성을 높입니다.

```typescript
function identity<T>(arg: T): T {
  return arg;
}

let value: string = identity("Hello, TypeScript"); // T가 string으로 유추됨
```

제네릭 유틸리티 타입:
- 제네릭 유틸리티 타입은 자주 사용되는 일반적인 제네릭 패턴을 미리 정의한 타입입니다. 예를 들어 `Partial<T>`, `Record<K, T>`, `Pick<T, K>`, `ReturnType<T>` 등이 있습니다. 이러한 유틸리티 타입은 코드를 간결하게 작성하고 타입 안정성을 유지하기 위해 유용합니다.

클래스의 Public, Private, Protected:
- **Public**: 기본적으로 클래스 멤버(속성 및 메서드)는 public으로 간주되며, 클래스 외부에서 접근할 수 있습니다.

```typescript
class Person {
  name: string; // public by default

  constructor(name: string) {
    this.name = name;
  }
}
```

- **Private**: private 멤버는 클래스 내부에서만 접근할 수 있습니다.

```typescript
class Person {
  private age: number;

  constructor(age: number) {
    this.age = age;
  }
}
```

- **Protected**: protected 멤버는 클래스 내부 및 서브클래스에서 접근할 수 있습니다.

```typescript
class Person {
  protected address: string;

  constructor(address: string) {
    this.address = address;
  }
}

class Student extends Person {
  constructor(address: string) {
    super(address);
    console.log(this.address); // protected 멤버 접근 가능
  }
}
```

클래스의 Static:
- `static` 키워드로 정적 멤버를 생성할 수 있습니다. 정적 멤버는 클래스의 인스턴스와 상관없이 클래스 자체에 속하며, 인스턴스

를 생성하지 않고도 접근할 수 있습니다.

```typescript
class MathUtils {
  static PI: number = 3.14159265359;

  static calculateCircumference(radius: number): number {
    return 2 * MathUtils.PI * radius;
  }
}

console.log(MathUtils.calculateCircumference(5)); // 정적 메서드 호출
console.log(MathUtils.PI); // 정적 속성 접근
```

위의 코드에서 `PI`와 `calculateCircumference`는 클래스 `MathUtils`에 속한 정적 멤버입니다.













# 리액트
React는 사용자 인터페이스를 만들기 위한 JavaScript 라이브러리로, Facebook에서 개발했습니다. React는 UI 컴포넌트를 구축하고 관리하기 위한 도구이며, 단일 페이지 애플리케이션(SPA) 및 웹 애플리케이션의 사용자 인터페이스를 만드는 데 주로 사용됩니다.

**React의 특징**:
1. **가상 DOM (Virtual DOM)**: React는 실제 DOM 대신 가상 DOM을 사용하여 UI 업데이트 성능을 향상시킵니다. 변경 사항을 가상 DOM에 먼저 적용하고, 이전 가상 DOM과 비교하여 실제 DOM에 필요한 최소한의 변경만 적용합니다.

2. **컴포넌트 기반**: React 애플리케이션은 컴포넌트로 구성됩니다. 컴포넌트는 재사용 가능하며, 각각이 자체적으로 상태(state)와 라이프사이클 메서드를 가질 수 있습니다.

3. **단방향 데이터 흐름**: React에서는 데이터의 흐름이 단방향입니다. 상위 컴포넌트에서 하위 컴포넌트로 데이터를 props로 전달하며, 하위 컴포넌트에서 이벤트 콜백을 통해 상위 컴포넌트로 데이터를 전달합니다.

4. **재사용성**: 컴포넌트 기반 아키텍처와 모듈화된 구조를 통해 코드의 재사용성이 높습니다.

5. **JSX**: React에서 UI를 정의하기 위한 JSX(JavaScript XML)를 사용합니다. JSX는 JavaScript의 확장 문법으로, HTML과 유사한 구조를 가지며 JavaScript 코드와 함께 사용할 수 있습니다.

**React의 장단점**:
- **장점**:
  - 가상 DOM을 통한 뛰어난 성능 최적화.
  - 컴포넌트 기반 아키텍처로 모듈화와 재사용성이 뛰어남.
  - 커뮤니티와 생태계가 활발하며, 다양한 라이브러리와 도구 지원.
  - JSX를 통한 직관적인 UI 작성.
  - 단방향 데이터 흐름으로 예측 가능한 상태 관리.
  
- **단점**:
  - 초기 학습 곡선이 다소 가파름.
  - 프로젝트 규모가 커질수록 상태 관리가 복잡해질 수 있음. (Redux 등의 도구 필요)
  - SEO에 대한 고려가 필요한 경우 추가 설정이 필요.

**Virtual DOM**:
- Virtual DOM은 React에서 성능 최적화를 위한 핵심 개념 중 하나입니다. 가상 DOM은 실제 DOM의 가벼운 복사본으로 이해할 수 있습니다. React는 상태 변경이 있을 때마다 이전 가상 DOM과 현재 가상 DOM을 비교하고 변경된 부분만 실제 DOM에 적용합니다. 이렇게 하면 불필요한 DOM 조작을 최소화하여 성능을 향상시킵니다.

**Virtual DOM 작동 원리**:
1. 초기 렌더링: 컴포넌트가 초기에 렌더링되면 가상 DOM이 생성됩니다.
2. 상태 변경: 컴포넌트의 상태나 속성이 변경되면 새로운 가상 DOM 트리가 생성됩니다.
3. 비교: 이전 가상 DOM과 새로운 가상 DOM을 비교하여 변경된 부분을 찾아냅니다.
4. 패치: 변경된 부분만을 실제 DOM에 반영합니다.

**JSX**:
- JSX(JavaScript XML)는 React에서 UI를 정의하는 문법으로, JavaScript 코드 내에서 HTML과 유사한 구조를 사용할 수 있게 해줍니다. JSX는 React 엘리먼트를 생성하고 렌더링할 때 사용됩니다.

```jsx
const element = <h1>Hello, JSX!</h1>;
```

**엘리먼트와 컴포넌트의 차이**:
- 엘리먼트는 React 애플리케이션의 가장 작은 구성 요소입니다. JSX를 사용하여 엘리먼트를 생성하며, React 엘리먼트는 UI의 일부를 나타냅니다.
- 컴포넌트는 엘리먼트의 집합으로, 재사용 가능한 UI 조각입니다. 함수형 컴포넌트 또는 클래스형 컴포넌트로 정의할 수 있으며, UI와 동작 로직을 함께 포함할 수 있습니다.

**React에서 컴포넌트 생성**:
- 함수형 컴포넌트:
  ```jsx
  function MyComponent(props) {
    return <div>Hello, {props.name}</div>;
  }
  ```

- 클래스형 컴포넌트:
  ```jsx
  class MyComponent extends React.Component {
    render() {
      return <div>Hello, {this.props.name}</div>;
    }
  }
  ```

**클래스형 컴포넌트와 함수형 컴포넌트의 차이**:
- 클래스형 컴포넌트는 `class`로 정의되며, 라이프사이클 메서드 및 상태(state)를 사용할

 수 있습니다.
- 함수형 컴포넌트는 함수로 정의되며, React Hooks를 사용하여 상태와 라이프사이클 기능을 활용할 수 있습니다. 함수형 컴포넌트는 코드가 간결하고 이해하기 쉽다는 장점이 있습니다.

**라이프사이클**:
- 라이프사이클은 컴포넌트가 생성, 업데이트, 제거되는 일련의 단계입니다. 클래스형 컴포넌트에서 주로 사용되며, 컴포넌트의 상태 변화나 DOM 조작을 다루는 데 유용합니다.

**라이프사이클 메서드**:
- `componentDidMount`: 컴포넌트가 처음으로 렌더링된 후에 호출되는 메서드. 초기 데이터 로딩 등에 사용됩니다.
- `componentDidUpdate`: 컴포넌트가 업데이트된 후에 호출되는 메서드. 업데이트된 데이터 처리에 사용됩니다.
- `componentWillUnmount`: 컴포넌트가 제거되기 직전에 호출되는 메서드. 리소스 정리에 사용됩니다.

**함수형 컴포넌트의 장점**:
- 간결한 문법과 이해하기 쉬운 코드.
- 상태 관리를 위한 React Hooks 제공.
- 더 빠른 성능 (일반적으로).

**React Hooks**:
- React 16.8 이후에 도입된 기능으로, 함수형 컴포넌트에서 상태(state)와 라이프사이클 특성을 사용할 수 있게 합니다. `useState`, `useEffect`, `useContext` 등이 포함됩니다.

**useEffect와 useLayoutEffect의 차이점**:
- `useEffect`: 컴포넌트 렌더링 후 비동기 작업을 수행하기 위해 사용됩니다. 렌더링 후에 작업을 수행하므로 화면이 먼저 그려진 후 작업이 실행됩니다.
- `useLayoutEffect`: 렌더링 직후(화면이 그려지기 전)에 작업을 수행합니다. 일반적으로 `useEffect`를 사용하며, 성능 최적화 목적으로만 `useLayoutEffect`를 사용해야 합니다.

**Pure Component**:
- `PureComponent`는 React의 성능 최적화를 위한 클래스형 컴포넌트입니다. `shouldComponentUpdate`를 자동으로 구현하여 상태나 속성이 변경되었을 때만 리렌더링됩니다.

**shouldComponentUpdate**:
- `shouldComponentUpdate` 메서드는 컴포넌트가 리렌더링되어야 하는지 여부를 결정합니다. 상태 또는 속성의 변경을 감지하고 필요한 경우 리렌더링을 방지할 수 있습니다.

**State**:
- State는 컴포넌트의 내부 상태를 관리하는 데 사용됩니다. 상태의 변경은 `setState` 메서드를 통해 이루어집니다.

**Props**:
- Props는 컴포넌트 간 데이터 전달에 사용되며, 부모 컴포넌트에서 자식 컴포넌트로 데이터를 전달할 때 사용됩니다.

**Props Drilling**:
- Props Drilling은 컴포넌트 계층 구조에서 여러 중간 컴포넌트를 통해 데이터를 전달해야 하는 상황을 묘사합니다. 이로 인해 코드가 복잡해지고 유지 관리가 어려워질 수 있습니다.

**Props Drilling 해결 방법**:
- Context API 또는 Redux와 같은 상태 관리 라이브러리를 사용하여 Props Drilling 문제를 해결할 수 있습니다. 이를 통해 중간 컴포넌트를 건너뛰고 데이터를 전달할 수 있습니다.

**데이터를 자식에서 부모로 전달**:
- React에서는 부모 컴포넌트에서 자식 컴포넌트로 데이터를 props로 전달하는 것은 기본적으로 지원합니다. 하지만 자식 컴포넌트에서 부모 컴포넌트로 데이터를 직접 전달하는 것은 복잡해질 수 있습니다. 이 경우 콜백 함수를 사용하여 데이터를 전달할 수 있습니다.

**Props와 State의 차이**:
- Props는 컴포넌트 간 데이터 전달에 사용되며, 부모 컴포넌트에서 자식 컴포넌트로 전달됩니다. 불변하며 읽기 전용입니다.
- State는 컴포넌트의 내부 상태를 관리하는 데 사용됩니다. 컴포넌트 내에서 변경할 수 있으며, `setState`를 통해 업데이트됩니다.

**setState 사용 이유**:
- React 컴포넌트의 상태를 변경할 때 `setState`를 사용해야 합니다. 이는 React가 내부적으로 상태 변경을 감지하고 컴포넌트를 다시 렌더링하도록 보장하기 위한 메커니즘입니다.

**React에서 상태 변화 감지 방법**:
- React는 상태 변화를 감지하기 위해 Virtual DOM과 Reconciliation(재조정) 프로세스를 사용합니다. 상태가 변경되면 React는 이전 가상 DOM과 새로운 가상 DOM을 비교하고 변경된 부분만을 실제 DOM에 반

영합니다.

**React에서 State의 불변성 유지**:
- React에서 상태의 불변성을 유지하기 위해 상태를 직접 수정하지 말고 `setState`를 사용해야 합니다. 이렇게 하면 React가 상태 변화를 감지하고 업데이트를 관리할 수 있습니다.

**setState 동기적 또는 비동기적 동작**:
- `setState`는 비동기적으로 동작합니다. 상태 업데이트는 지연될 수 있으므로 다음 상태에 의존하는 작업을 수행할 때 주의해야 합니다.

**비동기적 동작 이유**:
- React는 여러 `setState` 호출을 하나의 업데이트로 배치 처리하여 성능을 최적화합니다. 이로 인해 비동기적인 동작이 발생할 수 있습니다.

**HTML과 React의 이벤트 처리 차이**:
- React에서 이벤트 처리는 JavaScript 함수로 이루어집니다. HTML에서는 문자열로 이벤트 핸들러를 정의하는 것과 달리 React에서는 함수를 직접 전달합니다.

**Key Props 사용 이유**:
- `key` 프로퍼티는 React가 엘리먼트의 변경을 식별하는 데 사용됩니다. 동일한 컴포넌트의 여러 엘리먼트를 렌더링할 때 고유한 `key`를 제공하여 React에 업데이트를 효율적으로 적용할 수 있습니다.

**Ref의 용도**:
- Ref는 React 컴포넌트에서 DOM 요소나 컴포넌트 인스턴스에 직접 접근할 때 사용됩니다. 주로 특정 DOM 요소에 접근하거나 컴포넌트의 메서드를 호출하는 데 사용됩니다.

**제어 컴포넌트와 비제어 컴포넌트의 차이**:
- 제어 컴포넌트는 입력 요소의 값(value)를 React의 상태(state)와 동기화시키는 컴포넌트입니다. 사용자 입력에 반응하여 React 상태가 업데이트됩니다.
- 비제어 컴포넌트는 입력 요소의 값이 React 상태와 동기화되지 않는 컴포넌트입니다. 주로 React의 ref를 사용하여 DOM 요소에 접근하고 관리됩니다.

**HOC (Higher-Order Components)**:
- HOC는 React 컴포넌트를 감싸는 패턴으로, 컴포넌트를 래핑하여 새로운 기능을 추가할 수 있게 합니다. 코드 재사용과 추상화를 촉진하는 데 사용됩니다.

**Context API**:
- Context API는 React 애플리케이션에서 전역 상태를 공유하고 전달하기 위한 방법을 제공하는 API입니다. 주로 컴포넌트 간 데이터 공유에 사용됩니다.

**React.Fragment**:
- `React.Fragment`는 불필요한 부모 요소를 생성하지 않고 여러 자식 요소를 그룹화할 수 있는 React 컴포넌트입니다.

**Portal**:
- Portal은 React에서 DOM 트리의 다른 위치로 자식 컴포넌트를 렌더링할 때 사용됩니다. 모달 창과 같이 DOM 구조의 특정 부분에 렌더링하고 싶을 때 유용합니다.

**에러 바운더리**:
- 에러 바운더리는 컴포넌트의 하위 계층에서 발생한 오류를 처리하고 오류 경계를 설정하는 React 컴포넌트입니다.

**메모이제이션**:
- 메모이제이션은 함수의 결과를 캐시하여 동일한 입력에 대한 중복 계산을 방지하는 기술입니다. React에서는 `useMemo`와 `useCallback` 훅을 사용하여 메모이제이션을 구현할 수 있습니다.











# 프론트엔드 전반
**브라우저 렌더링 과정**:

1. **HTML 파싱**: 브라우저는 HTML 문서를 파싱하고 DOM 트리를 구축합니다. 파싱은 시작 태그부터 끝 태그까지 읽어가며 엘리먼트 노드를 생성합니다.

2. **CSS 파싱**: HTML 파싱 중에 CSS 링크나 스타일 태그를 만나면, 브라우저는 해당 CSS 파일을 가져와 CSS 파싱을 시작합니다. 스타일 정보를 가지고 있는 CSS 규칙들을 파싱하여 스타일 규칙 트리를 생성합니다.

3. **DOM 트리와 스타일 트리 결합**: DOM 트리와 스타일 트리를 결합하여 렌더 트리(Render Tree)를 생성합니다. 렌더 트리는 화면에 표시되는 요소들의 집합으로, 렌더링에 필요한 정보를 가지고 있습니다.

4. **레이아웃(Layout)**: 렌더 트리를 기반으로 각 엘리먼트의 위치와 크기를 계산하여 브라우저 창 내의 실제 위치에 배치합니다. 이 단계에서 엘리먼트의 크기나 위치가 결정됩니다.

5. **페인팅(Paint)**: 레이아웃 단계에서 계산된 엘리먼트의 스타일과 위치에 따라 화면에 픽셀을 그리는 작업이 수행됩니다. 이 단계를 페인팅 또는 리페인트라고 합니다.

6. **리페인트와 리플로우 최적화**: 렌더링 엔진은 최소한의 리플로우와 리페인트를 위해 최적화 작업을 수행합니다. 변화가 없는 부분은 캐싱되어 성능을 향상시킵니다.

7. **화면에 표시**: 모든 계산이 완료되면, 브라우저는 화면에 내용을 표시합니다.

**브라우저 동작**:

브라우저는 사용자가 웹 페이지를 요청하면 다음과 같은 순서로 동작합니다:

1. **URL 해석**: 사용자가 주소를 입력하면, 브라우저는 해당 URL을 해석하고 서버로 요청을 보냅니다.

2. **DNS 조회**: 브라우저는 도메인 이름을 IP 주소로 변환하기 위해 DNS(Domain Name System) 서버에 요청을 보냅니다.

3. **서버 연결**: 브라우저는 웹 서버와 TCP/IP 연결을 설정합니다.

4. **HTTP 요청**: 브라우저는 HTTP 요청을 서버로 보내고, 서버는 요청을 처리하여 응답을 생성합니다.

5. **응답 처리**: 브라우저는 서버로부터 받은 응답을 처리하고, HTML, CSS, JavaScript 등의 리소스를 다운로드합니다.

6. **HTML 파싱**: HTML 문서를 파싱하여 DOM 트리를 생성하고 렌더링 과정을 시작합니다.

7. **렌더링**: 앞서 설명한 브라우저 렌더링 과정을 따라 렌더링을 수행합니다.

8. **페이지 완료**: 렌더링이 완료되면 웹 페이지가 화면에 표시됩니다.

**Webpack**:
- Webpack은 모듈 번들러로, 프로젝트의 다양한 자원(JavaScript, CSS, 이미지 등)을 하나로 번들링하여 최적화된 번들 파일을 생성합니다. 모듈 시스템을 사용하여 코드를 모듈화하

고 관리할 수 있습니다.

**Babel**:
- Babel은 JavaScript 코드의 트랜스파일러로, 최신 버전의 JavaScript 문법을 이전 버전으로 변환하여 브라우저 호환성을 확보합니다. ES6+ 코드를 ES5로 변환하는 데 사용됩니다.

**Polyfill**:
- Polyfill은 구형 브라우저에서 지원하지 않는 JavaScript 기능을 추가하는 코드 조각입니다. 새로운 기능을 구현하거나 기존 기능을 수정하여 호환성을 유지합니다.

**ESLint**:
- ESLint는 JavaScript 코드에서 문법 오류나 스타일 가이드 룰을 검사하여 코드 품질을 향상시키는 도구입니다.

**Prettier**:
- Prettier는 코드 포맷터로, 일관된 코드 스타일을 적용하여 코드를 더 가독성 있게 만듭니다.

**SPA (Single Page Application)**:
- SPA는 하나의 HTML 페이지로 구성되며, 페이지 간의 전환이 발생할 때 JavaScript를 사용하여 필요한 부분만 동적으로 업데이트합니다. 전체 페이지를 다시 로드하지 않아 빠른 사용자 경험을 제공합니다.

**MPA (Multi-Page Application)**:
- MPA는 여러 개의 HTML 페이지로 구성되며, 각 페이지는 독립적으로 서버에서 렌더링되어 클라이언트에 전송됩니다. 각 페이지 간에는 완전한 요청과 응답이 이루어집니다.

**CSR (Client-Side Rendering) vs. SSR (Server-Side Rendering)**:
- CSR은 브라우저에서 JavaScript로 페이지를 렌더링하며 초기 로딩 속도가 빠르지만 SEO 문제와 초기 로딩 대기 시간이 발생할 수 있습니다.
- SSR은 서버에서 페이지를 렌더링하여 브라우저에 보내므로 초기 로딩 속도가 빠르고 SEO에 유리하지만 서버 부하가 높을 수 있습니다.

**CORS (Cross-Origin Resource Sharing)**:
- CORS는 웹 애플리케이션이 다른 도메인의 리소스를 요청할 때 발생하는 보안 정책입니다. 웹 브라우저에서 동일 출처 정책(SOP)을 준수하도록 요청과 응답을 처리합니다.
- CORS 문제를 해결하려면 서버에서 헤더를 설정하거나 프록시 서버를 사용하여 요청을 전달할 수 있습니다.

**번들 사이즈 최적화**:
- 번들 사이즈를 줄이려면 코드 스플리팅(Code Splitting)을 사용하여 페이지별로 번들을 분리하고, 불필요한 의존성을 제거하고 트리 쉐이킹(tree shaking)을 활용하여 불필요한 코드를 제거합니다.

**쿠키, 세션, 웹 스토리지의 차이**:
- 쿠키(Cookie): 클라이언트 측에 저장되며 주로 세션 식별, 사용자 추적에 사용됩니다.
- 세션(Session): 서버 측에 저장되며 클라이언트 식별, 상태 유지에 사용됩니다.
- 웹 스토리지(Web Storage): 클라이언트 측에 저장되며 데이터를 키-값 쌍으로 저장합니다.

**주소창에 주소 입력 흐름**:
1. 사용자가 주소를 입력하면 브라우저는 DNS 조회를 통해 해당 도메인의 IP 주소를 가져옵니다.
2. 브라우저는 서버와 TCP/IP 연결을 설정하고 HTTP 요청을 보냅니다.
3. 서버는 요청을 처리하고 응답을 생성하여 브라우저로 보냅니다.
4. 브라우저는 응답을 받아 HTML을 파싱하고 렌더링하여 화면에 표시합니다.










# 네트워크
**웹 프로토콜**:
- 웹 프로토콜은 웹에서 데이터를 주고받기 위한 통신 규약입니다. 이 규약은 클라이언트와 서버 간의 상호작용을 정의하며 웹 리소스(HTML 문서, 이미지, 스크립트 등)를 요청하고 전달하는 방법을 제공합니다.

**HTTP (Hypertext Transfer Protocol)**:
- HTTP는 웹에서 가장 널리 사용되는 프로토콜로, 클라이언트와 서버 간의 텍스트 기반의 데이터 전송을 담당합니다. 주로 웹 브라우저와 웹 서버 간의 통신에 사용됩니다.

**HTTP의 GET vs. POST**:
- **GET**: 정보를 요청하는 데 사용되며, 주로 쿼리 문자열(query string)을 통해 데이터를 전송합니다. 데이터 길이에 제한이 있고 보안에 취약합니다. 캐싱 가능하므로 반복 요청 시 캐시에서 응답을 가져올 수 있습니다.
- **POST**: 데이터를 서버로 전송하는 데 사용되며, HTTP 본문(body)을 통해 데이터를 전송합니다. 데이터 길이에 제한이 없고, 보안적으로 안전합니다. 캐싱되지 않으므로 항상 새로운 요청을 만듭니다.

**HTTP의 PUT vs. PATCH**:
- **PUT**: 리소스를 업데이트할 때 사용됩니다. 요청 본문에 업데이트된 리소스를 전달하며, 요청 시 리소스를 완전히 교체합니다.
- **PATCH**: 리소스를 부분적으로 업데이트할 때 사용됩니다. 요청 본문에 업데이트된 부분만을 전달하며, 리소스를 일부만 수정합니다.

**HTTP의 상태 코드 (Status Code)**:
- HTTP 상태 코드는 서버의 응답 상태를 나타내며, 3자리 숫자로 표현됩니다. 몇 가지 예시:
  - 200 OK: 요청이 성공했고 응답 데이터를 포함합니다.
  - 404 Not Found: 요청한 리소스를 찾을 수 없음을 나타냅니다.
  - 500 Internal Server Error: 서버에서 오류가 발생함을 나타냅니다.

**HTTP와 HTTPS의 차이점**:
- **HTTP (Hypertext Transfer Protocol)**: 데이터가 암호화되지 않고 평문으로 전송되므로 보안에 취약합니다.
- **HTTPS (HTTP Secure)**: 데이터가 SSL/TLS 프로토콜을 사용하여 암호화되어 전송되므로 보안성이 높습니다. 주로 민감한 정보를 주고받는 웹사이트에서 사용됩니다.

**HTTPS 동작 방식**:
1. 클라이언트가 HTTPS로 액세스하려는 서버에 연결합니다.
2. 서버는 공개키와 인증서를 클라이언트에게 제공합니다.
3. 클라이언트는 서버의 공개키를 사용하여 세션 키를 암호화하고 서버에 전달합니다.
4. 서버는 세션 키를 사용하여 데이터를 암호화하여 클라이언트에게 전달합니다.
5. 클라이언트는 세션 키를 사용하여 서버에서 받은 암호화된 데이터를 해독합니다.

**TCP vs. UDP**:
- **TCP (Transmission Control Protocol)**: 신뢰성을 중요시하는 프로토콜로, 데이터 전송 중 손실이나 오류가 발생할 경우 다시 전송합니다.
- **UDP (User Datagram Protocol)**: 빠른 전송을 중요시하는 프로토콜로, 데이터를 전송하지만 손실이나 오류에 대한 복구 기능이 없습니다.

**TCP의 신뢰성 보장**:
- TCP는 신뢰성을 보장하기 위해 **3-Way Hand

shake** 연결 설정과 **4-Way Handshake** 연결 해제를 사용합니다.

**3-Way Handshake**:
1. 클라이언트가 서버에게 연결 요청(시작 요청)을 보냅니다.
2. 서버가 연결 요청을 수락하고 클라이언트에게 연결 수락 및 초기화 응답을 보냅니다.
3. 클라이언트가 연결 수락 응답을 받고 서버와 연결이 설정됩니다.

**4-Way Handshake**:
1. 데이터 전송이 완료되면 클라이언트가 연결 종료 요청을 보냅니다.
2. 서버가 연결 종료 요청을 받고 데이터를 모두 보낸 후 클라이언트에게 연결 종료 응답을 보냅니다.
3. 클라이언트가 연결 종료 응답을 받고 서버와의 연결을 종료합니다.
4. 서버가 연결 종료 응답을 받고 클라이언트와의 연결을 종료합니다.










# 알고리즘 및 자료구조
**스택(Stack)과 큐(Queue)의 차이**:

- **스택(Stack)**: 후입선출(LIFO, Last-In-First-Out) 구조를 갖고 있으며, 데이터를 넣거나 빼는 동작이 한쪽 끝에서만 일어납니다. 즉, 최근에 추가된 데이터가 가장 먼저 제거됩니다. 함수 호출 스택이나 뒤로 가기 버튼의 동작을 모델링하는 데 사용됩니다.

- **큐(Queue)**: 선입선출(FIFO, First-In-First-Out) 구조를 갖고 있으며, 데이터를 넣는 동작은 한쪽 끝에서, 데이터를 빼는 동작은 다른 쪽 끝에서 일어납니다. 대기열이나 작업 큐 등에서 사용됩니다.

**좋은 코드의 조건**:
- 가독성: 코드가 이해하기 쉽고 다른 사람들이 쉽게 유지보수할 수 있어야 합니다.
- 효율성: 코드가 효율적으로 동작하고 불필요한 연산을 피해야 합니다.
- 확장성: 코드가 쉽게 확장 가능하도록 모듈화되고 유연해야 합니다.
- 안정성: 코드는 예외 상황을 처리하고 오류를 방지해야 합니다.

**좋은 알고리즘의 조건**:
- 정확성: 알고리즘이 올바른 결과를 반환해야 합니다.
- 효율성: 알고리즘이 입력에 대해 효율적으로 동작해야 합니다.
- 메모리 사용: 적절한 메모리 사용을 유지해야 합니다.
- 확장성: 큰 데이터셋에서도 동작할 수 있어야 합니다.

**정확도가 동일한 경우 다음으로 고려해야 할 조건**:
- 성능: 알고리즘의 실행 시간과 공간 복잡도를 고려해야 합니다.
- 유지보수 용이성: 코드를 쉽게 이해하고 수정할 수 있어야 합니다.
- 팀 협업: 팀 내에서의 협업을 고려한 설계와 구현이 필요합니다.

**시간 복잡도가 낮지만 메모리를 많이 사용하는 경우 대처 방법**:
- 메모리 관리: 메모리 누수를 방지하고 메모리를 효율적으로 활용하도록 코드를 개선합니다.
- 알고리즘 최적화: 다른 알고리즘을 사용하여 메모리 사용을 최소화하려고 노력합니다.
- 데이터 구조 개선: 데이터를 효율적으로 저장하고 관리하기 위해 적절한 데이터 구조를 선택하거나 설계합니다.

**완벽한 보안을 갖출 수 있는 알고리즘**:
- 완벽한 보안은 존재하지 않습니다. 보안은 지속적인 노력이 필요하며, 공격자가 발견한 취약점을 수정하고 업그레이드하는 과정입니다. 현실적으로는 취약성을 최소화하고 보호 대책을 마련하는 것이 목표입니다.

**알고리즘 공부 방법**:
- 기초 알고리즘 이론 학습
- 알고리즘 문제 풀기
- 다른 사람의 코드 읽기
- 온라인 강의나 책을 활용한 학습

**알고리즘 문제 푼 횟수**:
- 개인의 학습 경험에 따라 다를 수 있으며, 문제를 푸는 빈도는 개발자의 목표와 관심에 따라 다를 수 있습니다.

**DFS와 BFS 어려웠던 부분**:
- 재귀를 사용하지 않고 반복적으로 구현하는 것이 어려울 수 있으며, 그래프나 트리의 순회 과정을 이해하기 어려운 부분이 있을 수 있습니다.

**DFS와 BFS에서 어려웠던 부분**:
- 재귀 호출과 스택을 이용한 DFS 구현, 큐를 이용한 BFS 구현이 어려울 수 있습니다.

**재귀를 사용하지 않

고 DFS와 BFS를 구현하는 방법**:
- 스택을 이용하여 DFS를 반복적으로 구현하거나 큐를 이용하여 BFS를 반복적으로 구현할 수 있습니다. 반복적인 방식은 보다 메모리 효율적일 수 있습니다.
